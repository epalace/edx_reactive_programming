href="http://alaska.epfl.ch/~dockermoocs/handouts-edx/reactive-protocols.zip" target="[object Object]">download the reactive-protocols.zip</a> handout archive file and extract it somewhere on your machine.</p> <p>In this <g class="gr_ gr_84 gr-alert gr_gramm gr_inline_cards gr_run_anim Punctuation only-ins replaceWithoutSep" id="84" data-gr-id="84">exercise</g> we implement an Actor guarding a single value and offering transactional access to read or modify it&mdash;a so-called <em>Transactor</em>. We simplify this task by first creating a Behavior decorator for <em>selective receive</em>, programming the Transactor in <g class="gr_ gr_48 gr-alert gr_spell gr_inline_cards gr_run_anim ContextualSpelling ins-del multiReplace" id="48" data-gr-id="48">Erlang style</g>.</p> <h2>Selective Receive</h2> <p>As discussed in the video lecture on supervision in Akka Typed, Behaviors can be decorated or embedded in additional functionality. In this first part of the <g class="gr_ gr_71 gr-alert gr_gramm gr_inline_cards gr_run_anim Punctuation only-ins replaceWithoutSep" id="71" data-gr-id="71">exercise</g> we create a decorator that allows the embedded behavior to defer processing a message by <g class="gr_ gr_72 gr-alert gr_gramm gr_inline_cards gr_run_anim Style multiReplace" id="72" data-gr-id="72">returning </g><code>Behaviors.unhandled</code><g class="gr_ gr_72 gr-alert gr_gramm gr_inline_cards gr_disable_anim_appear Style multiReplace" id="72" data-gr-id="72">;</g> the message will then be put into a <code>StashBuffer</code> and retried upon each behavior change until it is finally accepted.</p> <p>The interface of the decorator is a single function:</p> <div class="highlight class="gr_ gr_80 gr-alert gr_gramm gr_inline_cards gr_run_anim Style multiReplace" id="80" data-gr-id="80">The </g><code>bufferSize</code><g class="gr_ gr_80 gr-alert gr_gramm gr_inline_cards gr_disable_anim_appear Style multiReplace" id="80" data-gr-id="80"> argument</g> determines the capacity of the <g class="gr_ gr_81 gr-alert gr_gramm gr_inline_cards gr_run_anim Style multiReplace" id="81" data-gr-id="81">internal </g><code>StashBuffer</code><g class="gr_ gr_81 gr-alert gr_gramm gr_inline_cards gr_disable_anim_appear Style multiReplace" id="81" data-gr-id="81">;</g> when more unhandled messages are outstanding than fit into this buffer, the decorator shall raise <g class="gr_ gr_82 gr-alert gr_gramm gr_inline_cards gr_run_anim Style multiReplace" id="82" data-gr-id="82">a disallows buffering, essentially requiring all messages to be handled upon <g class="gr_ gr_79 gr-alert gr_gramm gr_inline_cards gr_run_anim Grammar only-ins replaceWithoutSep" id="79" data-gr-id="79">first</g> arrival.</p> <p>The decorator shall retry all outstanding messages after each accepted message, where retries shall occur in the order of message reception; after the first accepted retry, all remaining outstanding messages shall be <g class="gr_ gr_49 gr-alert gr_spell gr_inline_cards gr_run_anim ContextualSpelling ins-del" id="49" data-gr-id="49">retried</g> from the head of the buffer, i.e. the retry always starts from the head of the buffer.</p> <h2>Transactor</h2> <p>A Transactor is a single-valued cell that allows transactions to be performed on the contained value. are not exposed to the general public. These model the protocol between the Transactor itself and the child actor it creates for each session, see below.</p> <h3>Behavior</h3> <p>The Transactor starts out with the given initial value. Other actors can interact with this value by first establishing an exclusive session and then exchanging messages with that <g class="gr_ gr_103 gr-alert gr_gramm gr_inline_cards gr_run_anim Style multiReplace" id="103" data-gr-id="103"><g class="gr_ gr_96 gr-alert gr_gramm gr_inline_cards gr_run_anim Grammar multiReplace" id="96" data-gr-id="96">session&rsquo;s</g> </g><code>ActorRef</code><g class="gr_ gr_103 gr-alert gr_gramm gr_inline_cards gr_disable_anim_appear Style multiReplace" id="103" data-gr-id="103">.</g> The session commands are:</p> <ul> <li><code>Extract</code>: apply the given projector function to the current Transactor value (possibly modified by the current session) and return the result to the <g class="gr_ gr_104 gr-alert gr_gramm gr_inline_cards gr_run_anim Style multiReplace" id="104" data-gr-id="104">given </g><code>ActorRef</code><g class="gr_ gr_104 gr-alert gr_gramm gr_inline_cards gr_disable_anim_appear Style multiReplace" id="104" data-gr-id="104">;</g> if the projector function throws an exception the session is terminated and all its modifications are rolled back</li> <li><code>Modify</code>: calculate a new value for the Transactor by applying the given function to the current value (possibly modified by the current session already) and return the given reply value to the given replyTo <code>ActorRef</code>; if the function throws an exception the session is terminated and all its modifications are rolled back; the <code>id</code> argument serves as a deduplication identifier: sending <g class="gr_ gr_94 gr-alert gr_gramm gr_inline_cards gr_run_anim Grammar multiReplace" id="94" data-gr-id="94">the a</g> <code>Modify</code> command with <g class="gr_ gr_93 gr-alert gr_gramm gr_inline_cards gr_run_anim Grammar multiReplace" id="93" data-gr-id="93">an </g><code>id</code><g class="gr_ gr_93 gr-alert gr_gramm gr_inline_cards gr_disable_anim_appear Grammar multiReplace" id="93" data-gr-id="93"> previously</g> used within the current session will not apply the modification function but send the reply immediately</li> <li><code>Commit</code>: terminate the current session, committing the performed modifications and thus making the modified value available to the next session; the given reply is sent to the given <code>ActorRef</code> as confirmation</li> <li><code>Rollback</code>: terminate the current session rolling back all modifications, i.e. the next session will see the same value that this session saw when it started</li> </ul> <p><g class="gr_ gr_102 gr-alert gr_gramm gr_inline_cards gr_run_anim Style multiReplace" id="102" data-gr-id="102">The </g><code>sessionTimeout</code><g class="gr_ gr_102 gr-alert gr_gramm gr_inline_cards gr_disable_anim_appear Style multiReplace" id="102" data-gr-id="102"> parameter</g> sets an upper limit on the duration of <g class="gr_ gr_95 gr-alert gr_gramm gr_inline_cards gr_run_anim Grammar multiReplace" id="95" data-gr-id="95">each</g> single session; when an active session takes longer than this timespan then it will automatically be rolled back and the Transactor becomes available again for the next session.</p> <h3>Implementation</h3> <p>We implement the Transactor as one top-level actor with selective receive that handles <g class="gr_ gr_76 gr-alert gr_gramm gr_inline_cards gr_run_anim Style multiReplace" id="76" data-gr-id="76">incoming </g><code>Begin</code><g class="gr_ gr_76 gr-alert gr_gramm gr_inline_cards gr_disable_anim_appear Style multiReplace" id="76" data-gr-id="76"> messages</g>, spawning one child actor for each session; <g class="gr_ gr_77 gr-alert gr_gramm gr_inline_cards gr_run_anim Style multiReplace" id="77" data-gr-id="77">new </g><code>Begin</code><g class="gr_ gr_77 gr-alert gr_gramm gr_inline_cards gr_disable_anim_appear Style multiReplace" id="77" data-gr-id="77"> messages</g> are deferred while a session is ongoing. The depth of the selective receive buffer shall be 30 messages.</p> <p>The Transactor uses DeathWatch to supervise each session, rolling back its effects upon failure. Upon session <g class="gr_ gr_69 gr-alert gr_gramm gr_inline_cards gr_run_anim Punctuation only-ins replaceWithoutSep" id="69" data-gr-id="69">timeout</g> the child actor for the active session is terminated.</p> <p>Upon <g class="gr_ gr_52 gr-alert gr_gramm gr_inline_cards gr_run_anim Style multiReplace" id="52" data-gr-id="52">session </g><code>Commit</code><g class="gr_ gr_52 gr-alert gr_gramm gr_inline_cards gr_disable_anim_appear Style multiReplace" id="52" data-gr-id="52">,</g> the session&rsquo;s child actor sends a <code>Committed</code> message to the Transactor, informing it about the new value to be guarded and releasing the session such that the next session may be accepted.</p> <h3>Hints</h3> <ul> <li><g class="gr_ gr_65 gr-alert gr_gramm gr_inline_cards gr_run_anim Style multiReplace" id="65" data-gr-id="65">The </g><code>Committed</code><g class="gr_ gr_65 gr-alert gr_gramm gr_inline_cards gr_disable_anim_appear Style multiReplace" id="65" data-gr-id="65"> </g><g class="gr_ gr_66 gr-alert gr_gramm gr_inline_cards gr_run_anim Style multiReplace" id="66" data-gr-id="66"><g class="gr_ gr_65 gr-alert gr_gramm gr_inline_cards gr_disable_anim_appear Style multiReplace" id="65" data-gr-id="65">and</g> </g><code>RolledBack</code><g class="gr_ gr_66 gr-alert gr_gramm gr_inline_cards gr_disable_anim_appear Style multiReplace" id="66" data-gr-id="66"> messages</g> contain the <g class="gr_ gr_67 gr-alert gr_gramm gr_inline_cards gr_run_anim Style multiReplace" id="67" data-gr-id="67">session&rsquo;s </g><code>ActorRef</code><g class="gr_ gr_67 gr-alert gr_gramm gr_inline_cards gr_disable_anim_appear Style multiReplace" id="67" data-gr-id="67"> to</g> allow the Transactor to safely ignore the messages pertaining to an old session&mdash;recall that all actor messaging and also termination are asynchronous.</li> <li>Modifications during an ongoing session are not communicated to the Transactor, only the final value is transmitted <g class="gr_ gr_59 gr-alert gr_gramm gr_inline_cards gr_run_anim Style multiReplace" id="59" data-gr-id="59">upon </g><code>Commit</code><g class="gr_ gr_59 gr-alert gr_gramm gr_inline_cards gr_disable_anim_appear Style multiReplace" id="59" data-gr-id="59">;</g> this way the Transactor is guaranteed to never store an intermediate value and thereby violate transactional behavior.</li> </ul></div> </div> <script type="text/javascript"> (function (require) {
